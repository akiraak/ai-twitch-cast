# 配信遅延検知・リアルタイム復帰機能の検証

## ステータス: 検討中

## 背景

ライブ配信では「今起きていること」がリアルタイムで視聴者に届くことが前提。何らかの原因で遅延が蓄積した場合、**原因を問わず**リアルタイムに復帰する仕組みが必要。

「コメントキューが溜まった」はその一例に過ぎない。

## 遅延が発生しうる箇所（全レイヤー）

### レイヤー1: コンテンツ生成（Python側）
| 箇所 | 遅延原因 | 現状 |
|------|----------|------|
| AI応答生成 | Gemini API遅延 | ブロッキング、キャンセル不可 |
| TTS生成 | Gemini TTS API遅延 | ブロッキング、キャンセル不可 |
| コメントキュー | コメント連続 → 処理が追いつかない | 純粋FIFO、スキップなし |
| SE再生待ち | `sleep(duration + 0.3)` | ブロッキング |
| TTS再生待ち | `sleep(duration + 0.5)` | ブロッキング |
| トピック発話 | 自動発話中にイベント発生 | 中断不可 |

### レイヤー2: 映像・音声パイプライン（C#側）
| 箇所 | 遅延原因 | 現状 |
|------|----------|------|
| フレームキャプチャ | WGC → BGRA → NV12変換 | 前フレーム処理中ならドロップ（OK） |
| ビデオパイプ | FFmpegが読み取り遅い → パイプ満杯 | 8MBバッファ、ドロップあり（OK） |
| 音声キュー | エンコード遅い → キュー蓄積 | 100チャンク上限でドロップ（OK） |
| TTSキュー | TTS PCMが溜まる | 上限なし、クリア手段なし |

### レイヤー3: エンコード・配信（FFmpeg → Twitch）
| 箇所 | 遅延原因 | 現状 |
|------|----------|------|
| エンコード速度 | CPU/GPU負荷 → speed < 1x | 検知なし、対処なし |
| RTMP送信 | ネットワーク遅延・帯域不足 | FFmpegが自動バッファ |
| Twitch側 | トランスコード・CDN | 制御不可（2〜5秒は固定） |

### 現状の問題
- レイヤー2は既にドロップ機構がある → **概ねOK**
- レイヤー3（Twitch側）は制御不可
- **レイヤー1が最大の問題**: コンテンツ生成が遅延の主因で、回復手段がない

## APIコスト分析

### 1回あたりのAPI呼び出し
| 処理 | モデル | 呼び出し回数 |
|------|--------|-------------|
| コメント応答 | Gemini 3 Flash Preview + Gemini 2.5 Flash TTS | AI 1回 + TTS 1回 = **2回** |
| トピック発話 | Gemini 3 Flash Preview (+ Google Search) + TTS | AI 1回 + TTS 1回 = **2回** |
| トピックローテーション | Gemini 3 Flash Preview | AI 1回 = **1回** |
| イベント応答 | Gemini 3 Flash Preview + TTS | AI 1回 + TTS 1回 = **2回** |

### カットイン（割り込み）のコスト問題

割り込みが発生すると、**中断された発話のAI+TTS費用が丸ごと無駄になる**:

```
トピック発話:
  generate_topic_line() → 課金済み ★無駄
  synthesize()          → 課金済み ★無駄
  [再生開始 → 1.5秒後にコメント到着]
  interrupt()           → 再生中断
コメント応答:
  generate_response()   → 新規課金
  synthesize()          → 新規課金
→ 合計4回のAPI呼び出し（本来2回で済むところ）
```

### コスト最適化: TTS生成前にキューチェック

**割り込みの大半のコストはTTS生成で発生する**（AI応答 < TTS音声生成）。
TTS生成前にキューをチェックすれば、無駄なTTSコストを回避できる:

```
トピック発話:
  generate_topic_line() → 課金済み（テキスト生成は安い）
  ★ キューチェック → コメントあり！
  synthesize() → スキップ！（TTS課金なし）
  → コメント応答に切り替え
```

**TTS前チェック** を組み込むことで:
- AI生成1回分（テキストのみ、安い）は無駄になるが
- TTS生成1回分（音声、高い）は回避できる
- 割り込みコストを大幅に削減

### キュースキップのコスト効果

スキップされたコメントは**AI呼び出しもTTS呼び出しもゼロ**:
- 5件スキップ = 10回分のAPI呼び出し節約
- まとめ応答（5件→1回のAI+TTS）なら10回→2回に削減

### コスト観点でのアプローチ比較

| アプローチ | API呼び出し | コスト効率 | 備考 |
|-----------|------------|-----------|------|
| 現状（変更なし） | 全コメント×2回 | 基準 | 遅延蓄積 |
| カットイン（TTS前チェックなし） | +2回/割り込み | △ 悪化 | 中断された分が無駄 |
| カットイン（TTS前チェックあり） | +1回/割り込み | ○ 許容範囲 | TTS課金を回避 |
| キュースキップ | −2回/スキップ | ◎ 節約 | スキップ分が丸ごと節約 |
| まとめ応答 | N件→2回 | ◎ 大幅節約 | 5件を1回で処理 |

## 提案: 遅延検知 + 自動復帰の仕組み

### 基本原則

> **「今」に追いつけないなら、古いものを捨てて「今」に戻る**
> **無駄なAPI呼び出しを最小限にしつつ、リアルタイム性を優先する**

### 1. 遅延の検知

#### 方法A: コンテンツキューの深さ（Python側、即実装可能）
```python
# comment_reader.py
queue_depth = len(self._queue)
if queue_depth > 0:
    logger.info(f"[latency] コメントキュー: {queue_depth}件待ち")
```
- キューが溜まっている = コンテンツ生成が追いついていない
- 閾値: 2件以上で「遅延状態」

#### 方法B: FFmpegエンコード速度（C#側）
FFmpegのstderrに出る `speed=` 値をパースする:
```
frame= 1500 fps= 30 ... speed=0.95x
```
- speed < 1.0 = エンコードが追いつけていない → 映像品質を下げるか、解像度を下げる
- 現状: stderrはログファイルに書いているだけ（`LogStderrAsync`）、`speed` 値のパースなし

#### 方法C: 音声キューの深さ（C#側）
```csharp
// 既に _audioQueue.Count をログしている（10秒ごと）
// → 常時高水位ならFFmpegが追いついていない
```

### 2. 復帰アクション

遅延を検知したら、レイヤーに応じた対処を自動で行う。

#### コンテンツ層（Python側）

**a. TTS前キューチェック（コスト最適化の要）**
- トピック/イベントのAI応答生成後、TTS生成前にコメントキューを確認
- キューにコメントがあれば → TTS生成をスキップ → コメント処理に切り替え
- **AI生成1回分は無駄になるが、TTS課金を回避**

**b. 発話中断（カットイン）**
- TTS前チェックをすり抜けた場合（TTS生成中〜再生中にコメント到着）の安全網
- `SpeechPipeline` に `interrupt()` メソッド追加
- `asyncio.sleep(duration)` を `asyncio.Event` でキャンセル可能に
- C#側の `tts_clear` でTTSキューもクリア

**c. キュースキップ**
- コメントキューが閾値を超えたら古いものを捨てる
- スキップしたコメントは「○○さん、△△さんもコメントありがとう！」と短くまとめる選択肢
- **スキップされた分のAPI呼び出しが丸ごと節約**

**d. まとめ応答（コスト最適）**
- キューが溜まったら複数コメントを1回のAI呼び出し + 1回のTTSで処理
- 5件を1回で処理 → API呼び出し10回→2回（80%削減）
- キュースキップとの併用: 溢れた分はスキップ、残りをまとめ

**e. 短縮モード**
- 遅延中はAI応答の文字数上限を下げる
- SE省略
- TTS再生待ちを短縮

#### 音声層（C#側）

**f. TTS即時クリア**
- `FfmpegProcess.ClearTts()` メソッド追加
- Python側から `ws_request("tts_clear")` で呼べるように
- 再生中のTTS音声を即停止

#### エンコード層（C#側、将来的）

**g. 動的品質調整**
- speed < 1.0 が続いたら、FFmpegを再起動して低い品質設定に
- 現実的にはHWエンコーダ（NVENC/AMF）使用時はほぼ問題にならない

### 3. 段階的エスカレーション（コスト考慮版）

| 状態 | 条件 | アクション | コスト影響 |
|------|------|-----------|-----------|
| 正常 | キュー0〜1件 | 通常処理 | 基準 |
| 軽度遅延 | キュー2〜3件 | SE省略、短縮モード | 微減（SE分） |
| 中度遅延 | キュー4件以上 | まとめ応答（最新3件を1回で処理） | 大幅削減 |
| 重度遅延 | キュー6件以上 | スキップ（最新2件のみ） + まとめ | 最大削減 |
| 発話中にコメント | トピック/イベント発話中 | TTS前チェック → 必要ならカットイン | TTS前チェックで最小化 |

## 実装計画

### Phase 1: TTS前キューチェック + 発話中断
最もインパクトが大きく、コスト効率も良い。

1. **Python側**: TTS前キューチェック
   - `_auto_speak()` 内でTTS生成前にキューを確認
   - キューにコメントがあればトピック発話を中止 → コメント処理へ
   - AI呼び出し1回分は無駄になるが、TTS課金を回避

2. **C#側**: `tts_clear` アクション追加
   - `FfmpegProcess.ClearTts()` — TTSキュー + カレントチャンクをクリア
   - `HttpServer` の WebSocket action switch に追加

3. **Python側**: `SpeechPipeline.interrupt()`
   - `_interruptible_sleep()` — `asyncio.Event` で即キャンセル可能なsleep
   - `interrupt()` — C# tts_clear + lipsync_stop + speaking_end 送信
   - `_speak_impl` の `asyncio.sleep(duration + 0.5)` を置換

4. **Python側**: `CommentReader` — コメント到着時の割り込み
   - トピック/イベント発話中にコメント到着 → `interrupt()` 呼び出し
   - コメント応答中は割り込みしない（キュー順）

### Phase 2: キュースキップ + まとめ応答
1. `_process_loop()` でキュー深さチェック
2. 閾値超過時に古いコメントをスキップ
3. まとめ応答の検討（複数コメント→1回のAI+TTS）
4. スキップ数のログ出力

### Phase 3: 遅延監視ダッシュボード（オプション）
1. キュー深さ、処理時間をAPIで公開（`/api/status` に追加）
2. WebUIで遅延状態を可視化
3. FFmpeg speed= 値のパース・表示

## リスク

- **音声の不自然な途切れ**: TTS中断時に音がプツッと切れる → 短いフェードアウト（C#側で10ms分をゼロに向かって減衰）の検討
- **中断タイミング**: TTS生成中（Gemini API呼び出し中）は即座にキャンセルできない → TTS前チェックで大半を回避、すり抜けた分は生成完了後に破棄
- **視聴者体験**: スキップされたコメントの人が不快 → まとめ言及で緩和
- **まとめ応答の品質**: 複数コメントをまとめた応答がAI次第 → プロンプト調整で対処
