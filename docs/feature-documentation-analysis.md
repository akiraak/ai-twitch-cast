# 機能ごとのREADME検討結果

## 背景

「機能ごとにREADME.mdなどのファイルを作り、機能を修正するときにそのファイルを参照し、また実装後はそのファイルを更新するような仕組み」の実用性を検討した。

## 現状すでにある類似の仕組み

| 仕組み | 場所 | 誰が使う | 特徴 |
|--------|------|----------|------|
| **メモリファイル** | `.claude/projects/.../memory/` | Claude Codeのみ | 会話開始時に自動読み込み。overlay.md, avatar.md等 |
| **docs/speech-generation-flow.md** | `docs/` | Claude Code + 人間 | CLAUDE.mdで「必ず参照」指示。最も成功している例 |
| **CLAUDE.md** | ルート | Claude Code | アーキテクチャ・規約・チェックリスト |
| **plans/** | `plans/` | Claude Code + 人間 | 実装計画。完了後は参照頻度が下がる |

## speech-generation-flow.md が成功している理由

- **複数ファイルにまたがる複雑なフロー**（lesson_generator → speech_pipeline → broadcast.html → C#アプリ）
- コードだけでは全体像がつかめない
- CLAUDE.mdに「変更前に必ず参照」の指示がある → 実際に効いている

## 「全機能にREADMEを作る」の問題点

1. **大半の機能はコードを読めば十分** — `tts.py`、`db.py`、`twitch_chat.py`等は単一ファイルで完結しており、READMEが二重管理になるだけ
2. **メモリファイルと役割が被る** — overlay.md、avatar.md、tts-audio.md等が既にある。これを「機能README」に移行するとメモリシステムとの二重管理が発生
3. **メンテナンスコストが高い** — ドキュメントは腐る。コミットのたびに「全関連READMEを更新」を徹底するのは現実的に難しい
4. **CLAUDE.mdの参照指示が増えすぎる** — 「この機能を触る前にX.mdを読め」が10個以上あると、Claude Codeの実効性が下がる

## 実用的な判断基準

READMEが有効なケース:

- **複数ファイル・複数コンポーネントにまたがるフロー**（speech-generation-flowのようなもの）
- **コードを読むだけでは設計意図が分からない**場合

READMEが不要なケース:

- 単一ファイルで完結する機能
- コードのdocstring/コメントで十分な機能
- 既にメモリファイルでカバーされている機能

## 結論

**全機能にREADMEを一律に作るのは過剰**。現状の3層構造（CLAUDE.md + メモリ + 個別docs）は十分に機能している。

### 現行の運用方針（維持）

1. **複雑なマルチファイルフロー** → `docs/` に専用ドキュメントを作成（例: speech-generation-flow.md）
2. **Claude Codeが参照する実装詳細** → メモリファイル（`.claude/projects/.../memory/`）で管理
3. **プロジェクト全体の規約・構成** → CLAUDE.md
4. **新しい仕組みの導入は不要** — 現行ルール「機能を追加・変更・削除したら対応するメモリファイルも更新すること」がそのまま目的を果たしている
