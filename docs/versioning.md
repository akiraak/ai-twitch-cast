# バージョニングルール

ちょびのバージョン（`VERSION` ファイル）は、**視聴者から見た変化の度合い** で判断する。
技術的な変更ではなく「ちょびの成長段階」として扱う。

## バージョン基準

| 区分 | 基準 | 例 |
|------|------|-----|
| **MAJOR** (x.0.0) | 別人級の変化 | アバター変更、声の変更、キャラ設定全面刷新 |
| **MINOR** (0.x.0) | 新能力・目に見える変化 | 新機能、新表情、画面の大きな変更 |
| **PATCH** (0.0.x) | 調整・修正 | バグ修正、微調整、パフォーマンス改善 |
| **上げない** | 開発者のみ | ドキュメント、テスト、プランのみ |

## 半自動提案の仕組み

コミット時にちょびがバージョンアップを提案する半自動方式を採用している。

### フロー

1. `post-commit` hook がコミットの diff を確認
2. **DONE.md が変更されている** → 機能完了の可能性 → ちょびに提案依頼
3. DONE.md が変更されていない → ドキュメント/テストのみ → スキップ
4. ちょびが配信で「バージョン上げたほうがいいかも〜」と提案
5. 開発者が判断し、Claude Code に「バージョン上げて」と指示 → `VERSION` 更新 + コミット

### 技術的な詳細

- `post-commit` hook 内で `git diff HEAD~1 --name-only` を確認
- DONE.md が含まれていれば `POST /api/avatar/speak` に `event_type="バージョン提案"` で送信
- サーバー未起動時は静かにスキップ（配信していない時は提案不要）

## VERSION ファイル

- プロジェクトルートの `VERSION` ファイルに `MAJOR.MINOR.PATCH` 形式で記載
- 配信画面のバージョンパネルに表示される（`{version}` テキスト変数）
- 更新時は `VERSION` ファイルを書き換えてコミットするだけ
